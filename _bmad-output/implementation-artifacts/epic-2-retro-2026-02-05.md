# Epic 2 Retrospective: Navigation documentaire

Date: 2026-02-05

## Epic Summary

- **Stories delivered:** 2/2 (100%)
- **Stories:** 2.1 Arbre documentaire navigable, 2.2 Rendu Markdown & lien GitHub
- **FRs couverts:** FR11, FR12, FR13, FR33, FR34

## What Went Well

- **Arbre récursif robuste:** `DocumentTree.vue` gère bien les structures `_bmad-output/` variées — dossiers imbriqués, fichiers à la racine, noms longs. Le composant est compact et maintenable.
- **`@nuxtjs/mdc` pour le rendu:** `<MDC :value="content" />` gère nativement le frontmatter YAML (le strip du rendu), les tables, le code fencé et les listes imbriquées. Pas besoin de parser manuellement.
- **Bouton "Open in GitHub" contextuel:** Le lien est construit dynamiquement avec owner/repo/path — toujours correct grâce aux données injectées via `useRepoData`.
- **Prose styling dark mode:** `prose dark:prose-invert` via Tailwind Typography fonctionne sans configuration supplémentaire.

## What Could Be Improved

- **Pas de sidebar mobile:** Sur viewport < 640px, l'arbre et le contenu s'empilent verticalement. Un `USlideover` serait plus ergonomique sur mobile.
- **Pas de recherche dans l'arbre:** Pour les repos avec beaucoup de documents, il faudrait un filtre texte dans la sidebar.
- **Pas de cache mémoire du contenu sélectionné:** Si on navigue entre deux documents, le contenu est re-fetché à chaque fois (même s'il est en cache Supabase côté serveur).

## Key Insights

- MDC de `@nuxtjs/mdc` est suffisant pour un viewer read-only — pas besoin de `@nuxt/content` avec ses collections et sa complexité.
- Le multi-format BMAD (FR33) est géré naturellement : l'arbre montre ce qui existe, les fichiers manquants ne causent pas d'erreur. La tolérance vient du design (pas de schema rigide attendu).

## Action Items

1. **USlideover mobile** — remplacer l'arbre vertical par un drawer sur mobile
2. **Cache mémoire client** — stocker le contenu déjà chargé dans un Map réactif pour éviter les re-fetch
